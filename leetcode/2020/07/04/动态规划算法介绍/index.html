1.如何思考动态规划问题？ 一开始是不知道这道题可以动态规划的，因此一开始就是模拟题目 比如路径问题，选哪一个，任意一个i,j点应该如何达到?从i-1,j和i,j-1到达 比如盗窃问题，选或不选，对于任意一个房间i,要么选择i-1，然后选择i和i-2 （自顶向下和自底向上无所谓）比如120三角形问题，选哪一个，从任意一点i,j出发，到达底边有两个选择，要么选择i+1,j 要么选择i+1,j+1 或者任意一个点i,j是如何到达的呢？是从i-1,j和i-1，j-1中来的。

然后通过画树就会发现，选择是有重复计算的，因此同样的计算备份一次。

然后呢，栈空间是有限的，如何转换成迭代呢？ 以120举例，已知递归法是从根结点递归到叶子节点， “递“是由函数栈出入栈，”归“是到叶子节点回退。因此可以直接去掉”递”，保留”归”，递是从根节点，那么归就是从叶子节点了。因此迭代法是和递归相反的方向。

如何写代码呢？ 如果状态转移是dfs(i,j) = dfs(i-1,j) + dfs(i,j-1)那么可以把等式两边+1，避免下标为负数，但是值得一提的是，这里更改的是memo的数组长度，nums数组长度不受影响， 保证nums数组长度不会被超出。

3.如何把递归改成递推呢？去掉递，合并归，树的下面开始往上计算，也就是为什么有些题目是递归是从数组末尾，递推是从0开始的 然后为了避免一些索引复数，可以通过两边加一的方式把状态转移方程平衡一下

1.当前操作是第i个元素的操作，子问题是前i个元素的状态和。memo数组是从一些元素中算出的结果，而不是从一个元素中

动态规划算法

动态规划算法是一个工具，帮助你递归更加的高效。因此，碰到一个问题，不是一开始就一头扎进动态规划， 而是先尝试解决这个问题，然后再去看看，有没有什么办法可以让其更加的高效。

动态规划算法核心点

  1. 状态如何定义
  2. 递归暴力解
  3. 重复子问题
  4. 最优子结构
  5. 迭代优化解
  6. 有后效性和无后效性 能用动态规划解题的一个特点是无后效性，也就是已经求解的子问题，不受到后续阶段的影响。 比如dp[i]=100，那么在算dp[i+1]的时候，dp[i]是不应该改变的，如果dp[i]会变，那就无法逐次递推推出了，也就不是动态规划的题目了。 李煜东著《算法竞赛进阶指南》，摘录如下： 为了保证计算子问题能够按照顺序、不重复地进行，动态规划要求已经求解的子问题不受后续阶段的影响。 这个条件也被叫做「无后效性」。换言之，动态规划对状态空间的遍历构成一张有向无环图，遍历就是该有向无环图的一个拓扑序。 有向无环图中的节点对应问题中的「状态」，图中的边则对应状态之间的「转移」，转移的选取就是动态规划中的「决策」。

Step1 暴力递归

第一步，我们先不管这道题能不能用动态规划解决，我们先尝试解决这两个问题，我们先从暴力解法。

以斐波那契数列为例，你想要知道F(5)第五个数是多少:

  1. 你得首先知道F(4)和F(3)
  2. 然后你又得知道F(2),F(1),F(0)
  3. F(1)和F(0)等于1和0

以不同路径的缩小版为例，想要抵达终点，只能从A或者B达到:

  1. A只能一个办法抵达终点（因为只能向右或者向下）
  2. B有两个选择C或者D，而C或者D只有一个选择抵达终点（因为只能向右或者向下）
  3. 因此可以发现，从起点到第一行或者第一列的每一个格子，只有一条路径。
  4. 递归法中状态定义：从起点到某个格子的路径

结论：

  1. 首先你会发现这个过程是递归的。
  2. 其次你会发现这个递归终止是有条件的，到F(1)或者F(0)或者某一个条件就会停下来。
  3. 最后，你会发现递归的过程是自顶向下的，也就是从不知道问题的解(F5)，到已知的解(F0或者F1)（相比较下文的迭代）

Step2 递归+记忆化搜索

如果把斐波那契数列的递归过程，画成一棵树的话，你会发现，有的枝叶是重复计算的。

如果把不同路径的缩小版的递归过程，画成一棵树的话，你会发现，有的枝叶是重复计算的。

同时，我们也可以从刚刚的递归中发现两个问题：

  1. 重复子问题（你在重复的解决子问题，上图的红色框）
    1. 由于递归是至顶向下的，因此算F(n)和F(n-1)的时候，必然是有重复算某一个值的情况，例如重复算F(2)
  2. 最优子结构（通过找到子问题的最优解决方案，你几乎免费地提取整个问题的最优解决方案）

因此，现在任务变成了要优化刚刚的暴力解法。增加一个数组memo，一旦memo中的值存在，我们就不再递归，直接返回。

虽然解法是对的，但是会引入新的问题，当输入是一个很大的值的时候，由于递归层数太多，超时了。

Step3 迭代+表格法

迭代往往不会带来空间或者时间上的好处，从递归转成迭代的原因是因为程序能够有更大的输入，计算更复杂情况的问题。因为我们常常会忽略程序递归调用栈的最大上限或者栈溢出（递归的实现是靠函数栈，函数栈所占有的内容不是无穷大的）。

把程序从递归改成迭代，往往需要表格法。

同时，要注意的是，表格法是自底向上的，为什么呢？

  1. 递归可以把未知问题递归到已知，62中的状态定义是从起点到某个格子的路径。
  2. 但是迭代问题在一个函数空间，要从已知值像滚筒一样滚向未知，没办法先计算终点的路径（不像递归）

Step4: 迭代+表格法（优化）

这是我最喜欢的一步，我们已经建立了一个MVP，然后我们在现有的代码上做优化。不是从0到1（这是最难的），而是从1到100。

往往动态规划的优化是可以在空间复杂度上进行优化。（以后碰到了时间优化可以举例子）
